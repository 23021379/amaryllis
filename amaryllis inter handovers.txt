### **1.0 Core Architectural Philosophy: The Pillars of Paranoia**

The Amaryllis pipeline is governed by a philosophy of extreme defensive design. We assume all external data is hostile, inconsistent, and unreliable. Our architecture is built to conquer this chaos.

1.  **Uncompromising CRS Unification:** All geospatial data, without exception, is immediately and forcibly re-projected to our internal standard, **EPSG:27700 (British National Grid)**, upon ingestion. This is the first and most critical gate, preventing the silent, catastrophic location errors that destroyed rival models.
2.  **Absolute Temporal Integrity:** We do not use information from the future to predict the past. All feature synthesis is performed using a "point-in-time" or "time-slice" methodology, ensuring that for any given application, the features are generated only from historical data that would have been available on its submission date.
3.  **Proactive Schema Normalization:** No external data is processed in its native format. As the immediate first step upon ingestion, all column schemas are programmatically forced into a single, predictable standard (all-lowercase). This "Normalize-Then-Join" pattern eradicates an entire class of `KeyError` failures at the point of data fusion.
4.  **Semantic, Not Syntactic, Logic:** We do not depend on the fragile, exact syntax of external column names. We use a semantically-normalized pattern-finding function (`find_column_by_pattern`) to locate required data based on its meaning, making the pipeline resilient to cosmetic schema changes (e.g., `Total Decisions` vs. `total_decisions`).
5.  **Paranoid Type Casting & State Management:** We never trust default type inference. All data is ingested as strings, then explicitly and safely coerced to the required types (`errors='coerce'`). After any major transformation (e.g., a `pivot_table`), we explicitly restore the geospatial state of our data (`gpd.GeoDataFrame()`) to prevent state mismatch errors.
6.  **Defensive, Modular Design:** The pipeline is broken into discrete, single-purpose scripts and functions. Each module is responsible for a specific task, making the system debuggable, maintainable, and robust.

### **3.0 The Data Engineering Pipeline: A Script-by-Script Breakdown**

The pipeline is a sequence of scripts, each producing a versioned, immutable data artifact (`L1`, `L2`, etc.).

#### **Script 1: `ingest_repd.py`**
*   **Directive:** 001
*   **Purpose:** To perform the initial, brutal sanitization of the raw REPD planning data.
*   **Input:** Raw `REPD.csv`
*   **Output:** `REPD_Amaryllis_L1.csv`
*   **Mandates Executed:**
    *   **Defensive Loading:** Only a pre-defined subset of columns is loaded.
    *   **Paranoid Type Casting:** All columns are loaded as strings (`dtype=str`), then explicitly coerced to numeric/datetime with `errors='coerce'`.
    *   **Schema Normalization:** All column names are forced to `snake_case`.
    *   **CRS Unification (Preparatory):** `X/Y-coordinate` columns are renamed to `easting/northing`, embedding the CRS context into the schema.
*   **Failures Mitigated:** Prevents pandas type-inference errors, handles corrupted cells without crashing, and establishes the foundation for our unified CRS pipeline, directly avoiding the failure mode of the DeepSeek model.

#### **Script 2: `feature_engineer.py`**
*   **Directive:** 002 & 004 & 005 (in its final evolved form)
*   **Purpose:** To generate target variables and a rich suite of first- and second-order behavioral features from the L1 data.
*   **Input:** `REPD_Amaryllis_L1.csv`
*   **Output:** `REPD_Amaryllis_L3.csv` (Note: L2 was an intermediate step)
*   **Mandates Executed:**
    *   **Definitive Target Generation:** Creates `permission_granted` and `planning_duration_days` with unambiguous logic.
    *   **Advanced Feature Engineering:** Decomposes dates into cyclical features (`sin`/`cos`); synthesizes advanced LPA metrics like workload trends, volatility, major/minor leniency gaps, enforcement ratios, and compliance rates.
    *   **Auditable Categorical Encoding:** Converts all string columns to integer codes and, crucially, exports a `categorical_mappings.json` file to make the encoding reversible and production-ready.
*   **Failures Mitigated:** Avoids the "architecturally hollow" feature set of the Claude model by creating nuanced, predictive signals. The JSON map ensures the model is not a "black box" and can be used on new, unseen data.

#### **Script 3: `create_knn_features.py` (Supersedes `create_temporal_features.py`)**
*   **Directive:** 007 (overriding 006)
*   **Purpose:** To create dynamic, temporally-correct, local precedent features for every solar application.
*   **Input:** `REPD_Amaryllis_L3.csv` (or L4 in the corrected version)
*   **Output:** `REPD_Amaryllis_L5.csv`
*   **Mandates Executed:**
    *   **Absolute Temporal Integrity:** This script is the embodiment of this principle. It uses an iterative, "time-slice" KNN engine.
    *   **The Temporal Guard:** For each target application, it builds a spatial index *only* on historical precedents whose final decision date was *before* the target's submission date.
*   **Failures Mitigated:** Prevents the most dangerous form of data leakage: using future knowledge to predict the past. This architecture ensures our model's backtest performance is a true and valid measure of its potential.

#### **Script 4: `enrich_grid_futures.py`**
*   **Directive:** 008
*   **Purpose:** To integrate quantitative data on grid capacity (headroom) from the DFES dataset.
*   **Input:** `REPD_Amaryllis_L5.csv`, `DFES.geojson`
*   **Output:** `REPD_Amaryllis_L6.csv`
*   **Mandates Executed:**
    *   **Uncompromising CRS Unification:** Immediately re-projects the DFES data from Lat/Lon (EPSG:4326) to BNG (EPSG:27700).
    *   **Temporal Paradox Mitigation:** Pivots the DFES data and uses a dynamic helper function (`get_relevant_headroom_col`) to select the correct forward-looking forecast based on each solar application's submission year.
    *   **Geospatial State Restoration:** Explicitly re-casts the pivoted data back into a `GeoDataFrame` to prevent state mismatch errors.
*   **Failures Mitigated:** Prevents both CRS contamination and the temporal paradox of using future grid forecasts to predict past outcomes.

#### **Script 5: `enrich_grid_stability.py`**
*   **Directive:** 009 & 010 (in its final evolved form)
*   **Purpose:** To integrate quantitative data on grid stability (fault levels).
*   **Input:** `REPD_Amaryllis_L6.csv`, `earthing_fault_level.csv`, `DFES.geojson`
*   **Output:** `REPD_Amaryllis_L7.csv`
*   **Mandates Executed:**
    *   **The "Geospatial Bridge":** Solves the non-geospatial data problem by using the DFES data as a location lookup to geolocate the fault level metrics.
    *   **Proactive Schema Normalization:** As the definitive project standard, it forces the schemas of both the fault level CSV and the DFES GeoJSON to lowercase *immediately upon ingestion* and *before* attempting to join them, guaranteeing key harmonization.
    *   **Aggregation by Maximum Constraint:** Aggregates node-level fault data using `max()` to model the true, worst-case engineering constraint for a connection.
*   **Failures Mitigated:** Solves the "missing data" problem for non-geospatial sources and permanently eradicates schema mismatch `KeyError` failures at the point of data fusion.

4.  **From Modularity to Artifact Integrity:** The failure to persist an L1 artifact for the substation data created a significant architectural debt that threatened to derail the project.
